# Portfolio Website

This context defines the language for Brian Hsu's personal website as a focused professional artifact, not a generic resume site or product marketing clone.

## Language

**Proof-of-work portfolio**:
A personal website whose primary purpose is to demonstrate credible technical ability through concrete work, artifacts, and judgment.
_Avoid_: Personal playground, fake product site, generic resume site

**Senior technical reviewer**:
The primary visitor: an experienced technical evaluator who judges Brian's systems and HPC ability by evidence, tradeoffs, and clarity of engineering judgment.
_Avoid_: Generic recruiter, casual visitor, product buyer

**Creative systems generalist**:
Brian's positioning thesis: an engineer who connects infrastructure, analytics, research, and interface design into coherent technical artifacts.
_Avoid_: Fast generalist, unfocused polymath, visual stylist

**Useful artifact world**:
The desired PostHog-inspired site quality: a playful desktop environment where each visible object leads to a real proof artifact such as a case study, demo, technical note, diagram, resume, or contact path.
_Avoid_: Decorative desktop, fake product navigation, empty folders

**Deep project case study**:
The first-class proof artifact: a substantial project page that explains the problem, constraints, architecture, tradeoffs, artifacts, and lessons behind a piece of work.
_Avoid_: Screenshot placeholder, shallow project card, resume bullet

**Capability domain**:
A top-level project grouping based on what kind of engineering ability the work proves, such as systems/infrastructure, analytics/data, research/ML, or interfaces/tools.
_Avoid_: Life category, chronology, generic folder

**Fictional artifact OS**:
The site's interaction shell: a custom operating-system-like environment where windows, files, icons, and maps organize Brian's proof artifacts.
_Avoid_: Mac clone, normal website with decoration, product OS clone

**Thesis plus proof paths**:
The required first-screen orientation: a concise positioning statement followed by a small set of obvious routes into the strongest proof artifacts.
_Avoid_: Desktop map only, hidden thesis, resume-only landing

**Infrastructure Systems**:
A primary proof path for work involving clusters, distributed systems, networking, operating environments, and production-like infrastructure.
_Avoid_: Systems as a catch-all

**Data & Analytics Systems**:
A primary proof path for work involving pipelines, measurement, operational analysis, dashboards, and data-informed decisions.
_Avoid_: Generic analytics, business metrics only

**Research/ML Tools**:
A primary proof path for computational tools that support research, machine learning workflows, scientific computing, or cognitive science exploration.
_Avoid_: Research as a life category

**Flagship proof artifact**:
The leading case study that should anchor the portfolio: Brian's HPC cluster build involving from-scratch cluster setup, node provisioning, networking, and research into monitoring and verifying InfiniBand and Ethernet behavior together.
_Avoid_: Template project, placeholder case study, unverifiable claim

**Sanitized architecture**:
A public technical artifact that explains architecture, constraints, provisioning flow, networking concepts, tradeoffs, and lessons while removing private identifiers, secrets, employer-specific topology, and sensitive operational details.
_Avoid_: Confidential implementation dump, vague high-level summary

**Provisioning/network verification**:
The core proof claim for the flagship case study: Brian can reason across bare-metal provisioning, cluster networking, measurement loops, system architecture, and the design choices that make the system understandable and verifiable.
_Avoid_: Installed a cluster, configured some nodes

**Shell plus proof**:
The visual hierarchy for the fictional artifact OS: the outer shell is a whimsical personal computer world, while the inner proof language uses tactile notes, diagrams, terminals, measurements, sketches, and case studies.
_Avoid_: Competing metaphors, decorative shell without proof, dense proof without personality

**Guided proof feed**:
The mobile presentation model: a readable linear path through the thesis, proof paths, flagship case study, artifacts, resume, and contact rather than a shrunken desktop interface.
_Avoid_: Mini desktop, hidden windows, unreadable mobile shell

**Credible scaffold**:
The near-term completeness standard: every visible object has a real label, real purpose, and honest preview, even when deep case-study content will be refined later.
_Avoid_: Fake completeness, template projects, dead placeholder surfaces

**Easter egg**:
A playful secondary object in the fictional artifact OS that can add personality but is not part of the primary proof path.
_Avoid_: Main navigation, dead utility, distracting decoration
